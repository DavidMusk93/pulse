# Task Execution Runtime Semantics

## Decision

Pulse task execution must not use a process execution timeout in the agent command protocol.

- `cmd.task_execute` and `cmd.shell_execute` must not carry `timeout_ms`.
- The agent must not kill task child processes because of a coordinator-provided timeout.
- `deadline_ms` may still exist on `PulseMessage` as a message lifecycle/dequeue deadline, but it is not a process execution timeout.
- Final task completion output must be lossless. The agent must not append `pulse output truncated` or drop final completion output bytes/chars.

## Rationale

Operational tasks may take highly variable time on loaded hosts. A hidden execution timeout makes long but healthy work look like task failure. The UI must display actual task state and output rather than a timeout generated by the control plane.

Large output should be handled by the existing chunked result protocol:

- Inline result stays small.
- Large encoded output uses `output_chunked=true` and `reply.task_result_chunk`.
- Coordinator reassembles chunks before exposing completion output.
- Running output streaming may still apply queue backpressure, but final completion output remains complete.

## Notes

Historical documents may mention `timed_out` or `timeout_ms`; treat those as legacy behavior unless a future design explicitly reintroduces bounded execution with user-visible controls.
